# Financial Filing RAG Agent

A focused V1 RAG pipeline for SEC 10-K annual report analysis with citation-style output.

The main demo corpus is JPMorgan Chase & Co.'s 2025 Form 10-K filing, filed on 2026-02-13. The project is designed as a financial research assistant that retrieves evidence from company filings and answers only from retrieved context.

This project follows the scope in [RAG_Project_Spec.md](/Users/ranxu/workspace/Rag-finance/RAG_Project_Spec.md) and currently supports:

- document ingestion
- text chunking
- embedding interface
- vector index build
- top-k retrieval with metadata-aware reranking
- grounded answer generation
- citation-style evidence output

## Highlights

- Focused corpus: a curated JPMorgan Chase 2025 Form 10-K dataset instead of a broad, noisy multi-source corpus
- Grounded output: answers are returned in `Answer / Sources / Evidence` format
- Inspectable retrieval: `--debug` exposes rerank score, section, topic, and quality metadata
- Metadata-aware ranking: retrieval combines vector similarity with SEC filing structure
- Demo-friendly CLI: the full pipeline can be shown with `build-index`, `ask`, and `eval`

## Why This Project

This repository is intentionally scoped as a small but explainable RAG system rather than a large production platform.

The goal is to demonstrate the core engineering pieces of a financial filing assistant:

- dataset preparation
- chunk schema design
- retrieval and reranking
- grounded generation
- evaluation and debugging

Using a single high-quality 10-K filing keeps the task bounded, improves answer traceability, and makes the system easier to evaluate and explain in an interview setting.

## Directory Responsibilities

The short version:

- `data/` contains files the pipeline reads or writes.
- `src/data/` contains Python code that knows how to read, validate, and transform those files.

Think of `data/` as the project's working material, and `src/data/` as the machinery that processes that material.

### `data/`: datasets and generated artifacts

`data/` should hold document content, intermediate outputs, and evaluation cases. These files are inputs or outputs of the RAG workflow, not reusable Python modules.

For GitHub, the repository keeps the curated JPM 10-K chunk file and JPM eval set, while large raw files and generated indexes are ignored. This keeps the interview demo reproducible without committing bulky local crawls or rebuild artifacts.

| Path | Purpose |
| --- | --- |
| `data/raw/finance_docs.jsonl` | Secondary raw document file consumed by `python3 main.py build-chunks`. Generated by `scripts/prepare_demo_dataset.py` from the larger article dump. |
| `data/raw/finance_article.jsonl` | Larger Sina Finance article dataset used as secondary source material. |
| `data/raw/finance_QA.jsonl` | Raw QA dataset used to prepare secondary evaluation cases. |
| `data/raw/jpm-20251231.htm` | Original JPMorgan 10-K HTML filing. |
| `data/raw/spec.md` | Human-readable chunking notes for the JPMorgan 10-K extraction experiment. It is documentation, not an input to the default CLI pipeline. |
| `data/processed/chunks.jsonl` | Secondary chunk output generated from `finance_docs.jsonl` by `python3 main.py build-chunks`. |
| `data/processed/jpm_2025_10k_chunks.jsonl` | Main demo corpus: specialized JPMorgan 10-K chunks generated by `scripts/prepare_jpm_10k_chunks.py`. |
| `data/eval/jpm_10k_eval_cases.json` | Main demo evaluation questions for JPM 10-K. |
| `data/eval/eval_cases.json` | Secondary evaluation questions prepared from the QA dataset. |

Ignored local files include raw crawls such as `data/raw/finance_article.jsonl`, `data/raw/finance_QA.jsonl`, the source filing HTML `data/raw/jpm-20251231.htm`, generated secondary chunks, and generated index files under `index/`.

### `src/data/`: data-processing code

`src/data/` should hold reusable code for the default pipeline's data layer.

| Path | Purpose |
| --- | --- |
| `src/data/schemas.py` | Defines the in-memory record shapes, currently `FinanceDocument` and `ChunkRecord`. |
| `src/data/loader.py` | Loads and validates JSONL files, and writes JSONL output. |
| `src/data/chunking.py` | Splits validated documents into chunk records using `CHUNK_SIZE` and `CHUNK_OVERLAP` from `config.py`. |

### `scripts/`: one-off or dataset-specific preparation

`scripts/` contains executable preparation utilities. These scripts may read large/raw files and write normalized files into `data/`.

| Script | Purpose |
| --- | --- |
| `scripts/prepare_demo_dataset.py` | Builds `data/raw/finance_docs.jsonl` from `data/raw/finance_article.jsonl`. |
| `scripts/prepare_eval_dataset.py` | Builds `data/eval/eval_cases.json` from `data/raw/finance_QA.jsonl`. |
| `scripts/prepare_jpm_10k_chunks.py` | Extracts selected sections from `data/raw/jpm-20251231.htm` and writes `data/processed/jpm_2025_10k_chunks.jsonl`. |

The main demo CLI path is:

```text
data/processed/jpm_2025_10k_chunks.jsonl
→ src/retrieval/
→ index/
→ src/pipeline/
```

## Data Format

The main JPM 10-K corpus is already chunked in `data/processed/jpm_2025_10k_chunks.jsonl`.

Each line is a JSON object like:

```json
{
  "id": "jpm_2025_10k_item1_0003",
  "company": "JPMorgan Chase & Co.",
  "doc_type": "10-K",
  "filing_date": "2026-02-13",
  "section": "Item 1 Business",
  "primary_topic": "business_segment",
  "secondary_topic": "segment_overview",
  "chunk_type": "text",
  "quality": "high",
  "title": "Business segments",
  "text": "...",
  "source": "data/raw/jpm-20251231.htm"
}
```

The secondary generic document pipeline still supports `data/raw/finance_docs.jsonl`.

Each line should be a JSON object like:

```json
{
  "doc_id": "doc_0001",
  "title": "央行降准释放长期流动性",
  "source": "Sina Finance",
  "date": "2026-04-10",
  "category": "macro",
  "content": "全文文本..."
}
```

Chunk output is written to `data/processed/chunks.jsonl` in this format:

```json
{
  "chunk_id": "doc_0001_chunk_0",
  "doc_id": "doc_0001",
  "title": "央行降准释放长期流动性",
  "source": "Sina Finance",
  "date": "2026-04-10",
  "chunk_index": 0,
  "text": "..."
}
```

For evaluation, the project also uses `data/eval/eval_cases.json`:

```json
[
  {
    "qid": "q1",
    "question": "...",
    "task_type": "qa",
    "reference_answer": "...",
    "source_dataset": "finance_QA.jsonl"
  }
]
```

`reference_answer` is included for manual comparison in this repository's V1 workflow.

## Quick Start

Build the main JPM 10-K retrieval index:

```bash
python3 main.py build-index --chunks data/processed/jpm_2025_10k_chunks.jsonl
```

Ask a JPM 10-K question:

```bash
python3 main.py ask "What are JPMorgan Chase's main business segments?"
```

Show retrieved chunks and rerank metadata during a demo:

```bash
python3 main.py ask "What are JPMorgan Chase's main business segments?" --debug
```

Show the prompt sent to the answer generator:

```bash
python3 main.py ask "What are JPMorgan Chase's main business segments?" --show-prompt
```

Run the JPM 10-K eval set:

```bash
python3 main.py eval --cases data/eval/jpm_10k_eval_cases.json
```

Example output:

```text
Answer:
Business segments: For management reporting purposes, the Firm has three reportable business segments - Consumer & Community Banking ("CCB"), Commercial & Investment Bank ("CIB") and Asset & Wealth Management ("AWM") - with the remaining activities in Corporate.

Sources:
- Business segments (data/raw/jpm-20251231.htm, 2026-02-13)

Evidence:
1. Business segments: For management reporting purposes, the Firm has three reportable business segments - Consumer & Community Banking ("CCB"), Commercial & Investment Bank ("CIB") and Asset & Wealth Management ("AWM") - with the remaining activities in Corporate.
```

Secondary data preparation commands:

```bash
python3 scripts/prepare_demo_dataset.py
python3 main.py build-chunks
python3 main.py build-index
python3 scripts/prepare_eval_dataset.py
python3 main.py eval
```

Compare hashing and real embedding retrieval on the JPM eval slice:

```bash
python3 scripts/compare_embedding_backends.py --limit 5
```

If `LLM_PROVIDER=openai` and a valid API key is configured, the CLI will use a real LLM backend automatically. If not, it falls back to the local grounded generator with no CLI change.

## Demo Workflow

For a quick walkthrough, the most useful sequence is to build the retrieval index, run one grounded question, inspect the retrieved evidence, and then run a short evaluation sweep.

1. Build the JPM 10-K index

```bash
python3 main.py build-index --chunks data/processed/jpm_2025_10k_chunks.jsonl
```

This prepares the retrieval index over the curated JPMorgan 10-K chunk set and stores chunk metadata for reranking and debugging.

2. Ask a question

```bash
python3 main.py ask "What are JPMorgan Chase's main business segments?"
```

This returns a citation-style answer constrained to retrieved filing context.

3. Inspect retrieval diagnostics

```bash
python3 main.py ask "What are JPMorgan Chase's main business segments?" --debug
```

This prints the retrieved chunks together with rerank score, embedding score, section, topic, and quality metadata.

4. Run a short evaluation sweep

```bash
python3 main.py eval --cases data/eval/jpm_10k_eval_cases.json --limit 3
```

This evaluates the same pipeline on a small curated JPM 10-K question set and shows answer output together with retrieval diagnostics.

For a shorter demo, steps 2 and 3 are usually enough to show the grounded answer path and retrieval transparency.

A static example report from a real run is available at [reports/sample_eval.md](/Users/ranxu/workspace/Rag-finance/reports/sample_eval.md).

For embedding-backend comparison, the repository also includes `scripts/compare_embedding_backends.py`, which generates `reports/embedding_comparison.md`.

## Retrieval And Reranking

The JPM 10-K retriever uses a two-stage strategy:

- vector retrieval over chunk text plus metadata
- metadata-aware reranking using section, topic, and quality labels

The reranking strategy follows [data/raw/spec.md](/Users/ranxu/workspace/Rag-finance/data/raw/spec.md):

```text
final_score =
0.70 * embedding_score
+ 0.15 * section_bonus
+ 0.10 * topic_bonus
+ 0.05 * quality_bonus
```

For example, business questions are biased toward `Item 1 Business`, while risk questions are biased toward `Item 1A Risk Factors` and `Item 7 MD&A`.

## Real Embeddings Configuration

The project can also use a real OpenAI-style embedding API for retrieval while keeping the same CLI.

Set environment variables like this:

```bash
export EMBEDDING_PROVIDER=openai
export EMBEDDING_API_KEY=your_api_key
export OPENAI_EMBEDDING_MODEL=text-embedding-3-small
export EMBEDDING_BASE_URL=https://api.openai.com/v1
```

Optional:

```bash
export OPENAI_EMBEDDING_DIMENSION=1536
export OPENAI_EMBEDDING_BATCH_SIZE=32
export EMBEDDING_TIMEOUT=60
```

`EMBEDDING_API_KEY` and `EMBEDDING_BASE_URL` are preferred for the embedding backend. If they are not set, the code falls back to `OPENAI_API_KEY` and `OPENAI_BASE_URL`.

Then rebuild the index with the same command shape:

```bash
python3 main.py build-index --chunks data/processed/jpm_2025_10k_chunks.jsonl
```

If the embedding dimension changes, the index must be rebuilt. The retriever checks index metadata and raises an explicit error when the query embedding dimension does not match the stored index dimension.

## Retrieval Backend

The code is written to prefer FAISS when available, which matches the spec. Chunk text and metadata are stored beside the vector index in SQLite, so retrieval storage is split into:

- vector search: FAISS when installed, otherwise the local simple vector backend
- chunk store: SQLite file at `index/index_store.chunks.sqlite3`

In the current local environment, `faiss` and `numpy` are not installed, so the project automatically falls back to a stdlib-only retrieval backend. This keeps the pipeline runnable while preserving the same top-level interfaces. Once FAISS is installed, the code can switch to the FAISS backend without changing CLI usage.

The answer generation layer prefers a real LLM when configured, but automatically falls back to a local grounded generator when no API key is available or the remote call fails. This keeps the repository runnable end-to-end while still making the demo stronger for GitHub and interview walkthroughs.

The embedding layer follows the same pattern: it prefers a real OpenAI-style embedding backend when configured, but falls back to the local hashing embedding baseline if no embedding configuration is available or the API request fails.

For experiments, the comparison script uses a strict remote-embedding path when configured, so the report can distinguish between an actual remote run and a skipped or failed remote backend.
